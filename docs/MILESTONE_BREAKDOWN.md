## 1. 文档目标

本文件把路线图进一步拆成可执行里程碑、阶段任务包、完成标准和风险提示。  
它不是代码任务列表，而是项目推进时的阶段拆解基线。

---

## 2. 里程碑总览

建议设置 6 个里程碑:

1. M0 规划定稿
2. M1 状态与数据基线建立
3. M2 预设与上下文闭环稳定
4. M3 参数与校准闭环稳定
5. M4 更新与发布体系稳定
6. M5 体验增强与可持续发版

---

## 3. M0 规划定稿

### 目标

- 全部基础规划文档齐备
- 项目硬约束冻结
- 后续开发方式统一

### 任务包

- 完成 PRD
- 完成 Architecture
- 完成 Data Model
- 完成 State Sync
- 完成 Module Refactor Plan
- 完成 Test Plan
- 完成 Release Process
- 完成 Roadmap 和 Milestone 文档

### 完成标准

- 所有基础文档进入版本控制
- 产品边界和非目标明确
- 后续阶段顺序明确

### 风险提示

- 如果 M0 未完成就继续大量改代码，后面会反复返工

---

## 4. M1 状态与数据基线建立

### 目标

- 建立统一状态真相源
- 建立当前上下文模型
- 减少页面层直接持久化

### 核心任务包

#### M1-A 状态盘点

- 盘点当前 `localStorage` key
- 盘点当前跨页面同步入口
- 盘点当前多窗口同步入口

产出:

- 状态清单
- 旧 key 映射表
- 高风险状态列表

#### M1-B Store 建立

- 建立 `UserConfigStore`
- 建立 `PresetSessionStore`
- 建立 `CalibrationStore`
- 建立 `UpdateStore`

产出:

- Store 接口定义
- 基础读写实现

#### M1-C 上下文解析服务

- 建立当前品牌 / 机型 / 版本 / 预设解析逻辑
- 清理 `_null` 上下文
- 统一当前预设识别逻辑

产出:

- `ContextResolver`
- 上下文异常策略

### 完成标准

- 关键状态不再散落在页面层
- 当前上下文可单独计算
- 多窗口同步有统一事件入口

### 验收用例

1. 切换机型后各页面上下文一致
2. 多窗口打开时当前配置能同步
3. 关闭引导状态在多窗口下保持一致

---

## 5. M2 预设与上下文闭环稳定

### 目标

- 让预设下载、显示、应用、识别形成稳定闭环
- 让本地、默认、远端预设边界清晰

### 核心任务包

#### M2-A 预设数据统一

- 统一预设元信息提取
- 版本识别改为基于内容而不是文件名
- 统一“当前应用”判定逻辑

#### M2-B 本地预设管理稳定

- 修正本地预设显示逻辑
- 修正本地与云端预设映射
- 修正 P1 / X1 等共享版本类型差异显示

#### M2-C 预设服务化

- 建立 `PresetService`
- 建立本地、默认、远端优先级规则
- 建立恢复默认的预设来源规则

### 完成标准

- 下载页、本地页、当前应用状态一致
- 预设版本识别不受文件名重命名影响
- 恢复默认有明确来源定义

### 验收用例

1. 重命名预设文件后“最新版本”判断仍正确
2. 本地有文件时下载页和本地页显示一致
3. 恢复默认后可回到安装包默认资源状态

---

## 6. M3 参数与校准闭环稳定

### 目标

- 让参数编辑、撤销重做、校准刷新形成稳定闭环
- 保证未保存保护和跨页面同步正确

### 核心任务包

#### M3-A 参数编辑服务化

- 建立 `ParamEditorService`
- 分离文本模式和分行模式的共享真相源
- 明确撤销 / 重做历史栈策略

#### M3-B 校准服务化

- 建立 `CalibrationService`
- 统一 Z / XY 值来源
- 参数变更后驱动校准页刷新

#### M3-C 草稿保护

- 建立未保存拦截
- 建立切页确认策略
- 建立外部修改提示策略

### 完成标准

- 参数页和校准页数值实时一致
- 保存和未保存状态明确
- 文本模式与分行模式不会各自维护独立内容

### 验收用例

1. 参数页修改 Z / XY 后校准页立即刷新
2. 未保存时切页会弹窗确认
3. 外部窗口修改当前预设时弹出刷新提示

---

## 7. M4 更新与发布体系稳定

### 目标

- 让更新可信
- 让回退可查
- 让发布工具与用户端分离

### 核心任务包

#### M4-A 更新链路稳定

- 固定本地版本源
- 固定远端 manifest 源
- 修正红点与手动检查逻辑重复

#### M4-B 回退链路稳定

- 明确 patch 应用失败处理
- 明确回退条件和日志
- 明确失败后 UI 提示

#### M4-C 发布工具剥离

- 从用户端构建中移除发布后台页面
- 建立独立发布工具入口
- 建立用户端构建白名单

### 完成标准

- 检查更新、下载、应用、回退闭环可验证
- 用户端默认不包含发布后台
- 用户安装包稳定 `<99MB`

### 验收用例

1. 手动检查更新能正确发现新版本
2. 应用补丁后重启能进入新版本
3. 补丁失败后能记录日志并回退
4. 用户安装包不包含后台页面与后台依赖

---

## 8. M5 体验增强与可持续发版

### 目标

- 在架构稳定基础上做高级体验
- 把用户端和开发者端都带入可持续维护状态

### 核心任务包

#### M5-A 首页与目录增强

- 品牌 / 机型展示模式优化
- 收藏、置顶、重命名、复制、图片管理体验优化

#### M5-B 预设管理增强

- 搜索、排序、多选、右键菜单体验细化
- 本地文件管理体验完善

#### M5-C 发版标准化

- 发布工具交互优化
- Markdown 发版说明流程优化
- 打包结果校验流程标准化

### 完成标准

- 体验增强不再反向破坏状态基线
- 发布流程可重复执行
- 开发者维护成本下降

---

## 9. 阻断条件

出现以下任一情况时，禁止进入下一个里程碑:

- 当前里程碑的验收用例未通过
- 高风险状态还没有唯一真相源
- 更新链路仍不可解释
- 用户安装包超过 `99MB`
- 发布后台仍在用户端默认构建里

---

## 10. 并行策略建议

可以并行的工作:

- 文档补齐与状态盘点
- schema 定义与 Store 设计
- 用户端更新链路梳理与发布工具剥离设计

不建议并行的工作:

- 参数页深改和更新系统深改同时进行
- 发布后台重构和用户端大规模 UI 改版同时进行
- 多个高风险领域同时迁移真相源

---

## 11. 里程碑之间的依赖关系

必须遵守以下依赖:

- M1 先于 M2、M3、M4
- M2 与 M3 可以部分交错，但都依赖 M1
- M4 必须在 M1 完成后再进入主线
- M5 必须建立在 M2、M3、M4 稳定基础上

原因:

- 没有状态基线，后面所有闭环都会反复抖动
- 没有更新与发布边界，包体和版本链路会继续失控

---

## 12. 最终执行建议

如果只保留一句执行建议，那就是:

先把“状态、预设、参数/校准、更新/发布”这 4 个领域做成稳定闭环，再做任何高级体验。

这是当前项目从“功能很多但容易炸”走向“真正可持续维护”的最短路径。
