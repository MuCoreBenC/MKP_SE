# 擦料塔预览坐标语义整改计划

## 背景

当前擦料塔预览存在一个让人误解的旧语义：

- `wiping.wiper_x / wiping.wiper_y` 在预览里不是“蓝色擦料塔方块左下角坐标”
- 预览使用了隐藏锚点与额外偏移量，导致视觉边距和用户输入的数值对不上
- 结果就是：
  - `X=5, Y=5` 时，用户会认为蓝色方块应距离 `0,0` 各 `5mm`
  - `X=152, Y=152` 且塔为 `20x20` 时，用户会认为右上边距应为 `8mm`
  - 但当前预览并不按这个逻辑显示，所以视觉上“8mm 反而比 5mm 更窄”

## 本次锁定的唯一语义

从这一轮开始，参数页里的擦料塔坐标必须满足下面这条唯一语义：

- `X / Y = 蓝色擦料塔方块左下角的真实坐标`

进一步约束如下：

- 蓝色方块就是用户真正关心的塔主体占地
- 蓝色方块的边距必须直接按它自己的外边缘到虚线边框去量
- 如果还存在额外安全包络，它只能作为限制逻辑或辅助提示存在
- 额外安全包络不能再让蓝色方块的视觉边距看起来失真

## 现状根因

目前代码同时混入了两套坐标参考：

- 预览层使用了隐藏锚点偏移
- 引擎层使用了模板中心点偏移
- 安全范围又混入了一个额外的隐藏包络

因此同一个 `wiper_x / wiper_y` 在三个地方的含义不一致：

1. 参数输入框里看起来像左下角坐标
2. 预览渲染时实际按隐藏锚点偏移
3. G-code 注入时又按模板中心偏移

## 目标结果

### 用户视角

- 输入 `X / Y` 时，蓝色方块必须准确落在这个左下角坐标
- 拖动蓝色方块时，输入框回写的也必须是左下角坐标
- `20x20` 默认塔的可见方块要比现在更大一些，更容易看、更容易拖
- 但“做大”只允许发生在 UI 的显示层，不允许悄悄改变真实塔尺寸

### 几何视角

- 可见占地：按塔主体真实宽高，从 `(X, Y)` 延伸到 `(X + width, Y + depth)`
- 隐藏安全包络：继续保留现有额外限制，但与可见占地分离
- 预览渲染：可见方块按真实占地画，安全包络只影响限制和提示

## TDD 执行清单

### Renderer / 预览

1. `X=5, Y=5` 时，蓝色方块左边缘距左侧虚线边框 `5mm`，底边距底部虚线边框 `5mm`
2. `X=152, Y=152` 且默认塔 `20x20` 时，蓝色方块右边缘距右侧虚线边框 `8mm`，上边缘距顶部虚线边框 `8mm`
3. 拖动蓝色方块后，输入框显示的必须是蓝色方块左下角坐标，而不是中心点或隐藏锚点
4. 放大蓝色方块的显示尺寸后，边距比例不能失真
5. 如果后续启用更大的 brim / 外墙 / 斜肋外墙，蓝色方块占地也要对应变大

### Engine / 坐标换算

1. `wiperX / wiperY` 必须和预览语义一致，表示塔主体左下角
2. 模板偏移量不能再写死成当前的旧锚点语义
3. 首层塔路径和普通层塔路径都必须使用同一套左下角语义
4. 安全范围仍可保留额外左右预留，但不能回流污染蓝色方块预览

## 实施步骤

1. 先补 smoke / unit TDD，锁死“左下角真实坐标”语义
2. 重构 renderer 的几何换算：
   - 可见方块偏移改为从 `0..width` / `0..depth`
   - 安全包络单独计算
3. 调整拖拽与输入框回写逻辑：
   - 统一按左下角坐标读写
4. 重构 engine 模板偏移：
   - 改成围绕“左下角锚点”做模板平移
   - 首层 / 普通层保持一致
5. 放大蓝色方块的 UI 显示尺寸，但不改真实几何
6. 跑相关测试，必要时补更多边界用例

## 非目标

本轮不做下面这些额外扩展：

- 不改用户实际 preset 文件结构
- 不引入新的塔类型
- 不引入新的拖拽交互模式
- 不顺手改其它无关 UI

## 验收标准

- 用户看到的 `X / Y` 与蓝色方块左下角位置完全一致
- 左下角 `5,5` 和右上角 `152,152` 的视觉边距符合直觉和计算
- 预览、拖拽、输入框、CLI 生成的坐标语义一致
- 相关测试全部通过
