## 2. 产品定义

### 2.1 产品名称

MKP SupportE

### 2.2 产品定位

这是一个面向 3D 打印用户的桌面软件，核心目标是:

- 帮助用户选择对应机型和版本预设
- 下载、管理、应用支撑面改善预设
- 可视化校准 Z/XY 偏移
- 编辑预设参数与 G-code 相关配置
- 通过后处理流程把预设应用到打印工作流
- 通过热更新和发布中心持续演进软件与预设资源

### 2.3 目标用户

- 普通 3D 打印用户
- Bambu / Anycubic / DIY 等机型用户
- 需要通过 GUI 完成配置而不想手动改 JSON 的用户
- 维护默认资源、版本、补丁包的项目维护者

### 2.4 产品形态

- Electron 桌面 GUI 主程序
- CLI / G-code 后处理模式
- 独立发布中心 / 默认资源后台

### 2.5 项目硬约束

以下约束视为后续所有设计与开发的硬边界:

- 面向用户发布的 Windows 安装包必须控制在 `99MB` 以内
- 国际化只保留中文和英文
- 不建设用户画像、行为埋点、云端分析平台
- 仅保留基础错误日志与故障诊断能力
- 发布中心 / 默认资源后台必须可从用户安装包中剥离

### 2.6 非目标澄清

当前阶段明确不做:

- 多语言平台化扩展
- 云端账号体系
- 云端用户行为分析
- A/B 实验平台
- 面向普通用户开放的内置发布后台

### 3.2 当前已实现能力

按代码观察，当前已经不是一个 Demo，而是一个功能较多、但工程化尚未稳定的中型桌面工具。已存在能力包括:

- 品牌与机型目录展示
- 默认目录数据与用户自定义目录覆盖
- 机型图片、收藏、置顶、复制、重命名、右键菜单
- 预设在线获取、本地预设应用、预设 manifest 回退
- Z 偏移可视化校准
- XY 偏移可视化校准
- 参数页动态表单编辑
- G-code 文本/分行双模式编辑
- 参数修改保存、恢复默认、撤销历史
- 当前预设缓存与跨页面刷新
- 版本页面、热更新检查、红点提醒、回退
- 发布中心:
  - 编辑 `package.json`
  - 编辑 `cloud_data/app_manifest.json`
  - 打包热更新补丁
  - 构建完整安装包
  - 管理默认品牌/机型/图片/预设
- CLI G-code 后处理引擎

### 3.3 当前核心问题

这个项目最大的问题不是“功能少”，而是“功能已经很多，但边界和质量控制还没有跟上”。目前最明显的问题有:

- `app.js` 仍然过大，且承担了过多跨域职责
- `home.js`、`updates.js`、`app.js` 存在历史重复实现与覆盖写法
- 仍有历史乱码和旧字符串残留
- 使用 `localStorage` 作为大量状态源，缺少明确的配置层与同步协议
- 运行时状态、页面状态、业务状态耦合严重
- 预设、目录、云端资源、默认资源、用户覆盖资源的边界还不够清晰
- 主程序和发布中心共用大量逻辑，但尚未做稳定的领域拆分
- 没有自动化测试
- 没有正式的需求基线、接口契约、数据 schema、迁移策略
- 发版流程虽然可用，但仍偏“工具化”，不是标准化流水线

## 4. 软件本质功能分析

从业务上看，这个软件不是单一工具，而是三套系统叠加:

### 4.1 用户端桌面工具

面向普通用户，完成:

- 机型选择
- 预设获取与应用
- 校准
- 参数修改
- 常见问题与设置

### 4.2 预设与资源管理系统

面向内容维护，完成:

- 默认品牌和机型目录管理
- 图片管理与转换
- 默认预设编辑
- 预设 manifest 管理

### 4.3 发布与更新系统

面向维护者，完成:

- 版本信息编辑
- 热更新补丁打包
- 全量安装包构建
- 云端 manifest 驱动的更新与回退

结论:

这个项目本质上已经不是“一个 Electron 小工具”，而是:

- 一个面向终端用户的专业桌面应用
- 一个面向维护者的轻量后台系统
- 一个带热更新、资源版本控制和默认资源治理的内容交付系统

## 7. 当前阶段的真实开发状态判断

### 7.1 项目成熟度判断

当前项目处于:

- 产品能力: 已进入 Beta 中后期
- 工程能力: 仍处于原型演进到工程化之间
- 发布能力: 初步可用，但还没达到标准化

### 7.2 当前最危险的开发方式

继续直接加功能、直接修 UI、直接补交互，而不先收口架构，会导致:

- 状态问题越来越多
- 多窗口/多页面联动问题持续反复
- 预设编辑和校准之间越来越难维护
- 更新、回退、默认资源、用户覆盖层边界越来越混乱
- 发布中心和用户端逻辑互相污染

因此，下一阶段严禁继续“看到哪里改哪里”的开发方式。

## 14. 风险清单

### 高风险

- 状态源过多导致配置覆盖
- 预设文件写入与页面状态不同步
- 更新链路误把远端版本当本地版本
- 发布中心修改默认资源后影响用户端稳定性
- 历史重复代码继续叠加

### 中风险

- 图片、预设、manifest 之间的映射失配
- 多窗口同时编辑同一预设
- 无 schema 导致文件格式漂移
- 无自动化测试导致回归成本持续上升

### 低风险

- 视觉样式继续迭代
- FAQ 与 about 文案维护

## 17. 最终结论

这个项目已经具备了做成一款成熟桌面软件的基础，但当前最大的瓶颈不是功能不够，而是:

- 缺少正式需求基线
- 缺少明确架构边界
- 缺少数据与状态治理
- 缺少自动化测试与标准发版流程

因此，正确策略不是继续直接堆功能，而是:

- 先完成需求冻结
- 再完成架构定稿
- 再做核心重构
- 然后按模块闭环推进功能

只有这样，后续新增机型、预设、发布中心、热更新、参数编辑这些复杂能力，才不会继续把项目拖回到“边修边炸”的状态。
