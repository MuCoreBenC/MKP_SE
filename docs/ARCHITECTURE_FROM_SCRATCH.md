# MKP SupportE 从 0 开始架构设计文档

版本: v1.0  
状态: 草案  
日期: 2026-03-15  
适用范围: 从 0 重新规划 MKP SupportE 的系统架构、模块边界、状态治理与发布体系

---

## 1. 文档目标

本文件用于定义从 0 重新设计 MKP SupportE 时的技术架构原则与落地方案，重点解决以下问题:

- 模块边界如何划分
- 数据如何分层
- 状态如何管理
- 多窗口如何同步
- 预设、默认资源、用户资源、云端资源如何隔离
- 更新与发布如何标准化

本文件与 `PRD_FROM_SCRATCH.md` 配套使用。

---

## 2. 架构设计原则

### 2.1 核心原则

1. 业务规则不能直接写在页面 DOM 逻辑里  
2. 所有持久化写入都必须经过服务层  
3. 默认资源、用户覆盖层、远端资源必须严格分层  
4. 多窗口同步必须有统一协议  
5. 高风险流程必须可回退  
6. 页面状态和业务状态必须分开  
7. 更新系统与运行时版本源必须隔离  
8. 发布中心与用户端共享能力，但不共享页面实现  
9. 所有关键数据结构必须有 schema  
10. 任何模块都不能依赖“重复定义最后覆盖生效”

### 2.2 非目标

- 不追求过早微前端化
- 不引入复杂服务端
- 不为了架构而架构
- 不引入过重框架，仅在必要时引入清晰的状态与模块化机制
- 不引入用户画像和行为埋点平台
- 不把发布后台直接耦合进终端用户安装包

### 2.3 架构硬约束

- 用户安装包大小必须长期稳定控制在 `99MB` 以内
- 语言资源仅支持 `zh-CN` 和 `en-US`
- 运行时仅建设本地日志、崩溃信息和手动导出诊断包
- 发布中心与默认资源后台必须支持独立构建、独立运行、独立分发

## 3. 系统上下文

### 3.1 系统组成

MKP SupportE 从架构上分为 5 个部分:

1. Electron 主进程
2. Electron 渲染进程
3. 本地文件资源层
4. 云端资源层
5. 发布构建层

最终交付时应拆成两类产物:

1. 用户端安装包
2. 开发者侧发布工具包

### 3.2 系统边界图

逻辑边界:

- 用户操作只进入渲染层控制器
- 渲染层控制器调用领域服务
- 领域服务通过 IPC 访问主进程
- 主进程负责文件、系统、打包、更新
- 云端访问统一由适配层管理

---

## 4. 总体分层架构

建议采用 5 层架构。

### 4.1 展示层

职责:

- 页面结构
- 视觉交互
- 动画
- 提示
- 基础组件

不负责:

- 文件读写
- 状态持久化
- 核心业务判断

### 4.2 页面控制层

职责:

- 组织页面行为
- 响应用户操作
- 调用服务层
- 绑定 store

建议页面控制器:

- `home-controller`
- `presets-controller`
- `calibration-controller`
- `params-controller`
- `updates-controller`
- `release-controller`
- `settings-controller`

### 4.3 领域服务层

职责:

- 承载真正业务规则
- 屏蔽页面细节
- 组合主进程能力与本地状态

建议服务:

- `CatalogService`
- `PresetService`
- `CalibrationService`
- `ParamEditorService`
- `UpdateService`
- `VersionService`
- `ReleaseService`
- `ThemeService`
- `OnboardingService`

### 4.4 状态层

职责:

- 管理业务状态
- 管理页面状态
- 管理持久化状态
- 提供订阅更新能力

建议 store:

- `UserConfigStore`
- `CatalogStore`
- `PresetSessionStore`
- `CalibrationStore`
- `ParamEditorStore`
- `UpdateStore`
- `ThemeStore`
- `ReleaseStore`

### 4.5 基础设施层

职责:

- IPC
- 文件系统
- 本地存储
- 网络请求
- schema 校验
- 日志与错误处理

建议适配器:

- `IpcBridge`
- `StorageAdapter`
- `FileAdapter`
- `ManifestAdapter`
- `RemoteFetchAdapter`
- `LoggerAdapter`

## 5. 推荐目录结构

从 0 开始时建议采用如下结构:

```txt
src/
  main/
    app/
      bootstrap/
      ipc/
      services/
      adapters/
      schemas/
      utils/
    main.js
  renderer/
    app/
      core/
      stores/
      services/
      controllers/
      sync/
      components/
      pages/
      adapters/
      utils/
      schemas/
    assets/
      css/
      images/
      icons/
    index.html
    release_center.html
```

说明:

- `assets/js` 不再作为所有逻辑堆积目录
- 页面逻辑与通用逻辑分离
- schema 单独收口
- sync 单独收口

## 6. 模块边界设计

### 6.1 Catalog 域

负责:

- 品牌和机型目录
- 默认目录
- 用户覆盖目录
- 图片导入与裁剪

不负责:

- 当前预设
- 校准值
- 参数编辑

### 6.2 Preset 域

负责:

- 预设列表
- 在线清单
- 本地清单
- 下载
- 应用
- 当前预设识别
- 预设文件管理

不负责:

- 参数编辑器内部历史
- 校准 UI 呈现

### 6.3 Calibration 域

负责:

- Z/XY 偏移提取
- 偏移写回
- 校准界面状态
- 校准前置条件判断

不负责:

- 在线预设下载
- 参数字段渲染

### 6.4 Param Editor 域

负责:

- 参数加载
- 参数 schema 映射
- 文本模式与结构化模式
- Undo / Redo
- Dirty 状态
- 保存与恢复默认
- 外部修改冲突检测

不负责:

- 机型目录
- 更新
- 发布

### 6.5 Update 域

负责:

- 当前版本识别
- 远端 manifest 获取
- 红点提醒
- 热更新检查
- 热更新应用
- 回退版本

不负责:

- 发布构建
- release notes 编辑

### 6.6 Release 域

负责:

- 版本信息编辑
- manifest 生成
- 补丁构建
- 安装包构建
- 默认资源管理

不负责:

- 终端用户运行时页面逻辑

## 7. 数据分层设计

### 7.1 数据来源划分

必须严格区分以下 5 类数据:

#### D1 默认代码数据

示例:

- 默认品牌与机型目录定义

特点:

- 跟随安装包
- 可被发布中心重写

#### D2 默认资源数据

示例:

- `cloud_data/presets`
- `cloud_data/app_manifest.json`
- `cloud_data/presets_manifest.json`

特点:

- 跟随安装包
- 作为离线回退源

#### D3 用户覆盖层数据

示例:

- 用户自定义品牌/机型目录
- 用户导入的图片
- 用户本地预设文件

特点:

- 不覆盖默认安装资源
- 与安装包解耦

#### D4 运行时状态

示例:

- 当前页面
- 当前机型
- 当前版本
- 当前已应用预设
- 编辑器未保存状态

特点:

- 一部分持久化
- 一部分只在内存里存在

#### D5 远端数据

示例:

- app manifest
- presets manifest
- patch 包

特点:

- 只读
- 不可直接当成本地安装状态

## 8. Schema 设计要求

### 8.1 必须定义的 schema

- `user-config.schema.json`
- `catalog-brand.schema.json`
- `catalog-printer.schema.json`
- `preset.schema.json`
- `presets-manifest.schema.json`
- `app-manifest.schema.json`
- `release-config.schema.json`
- `param-field-meta.schema.json`

### 8.2 schema 使用原则

- 读文件先校验
- 写文件前校验
- schema 不通过不能写入
- schema 变更必须带版本号和迁移逻辑

---

## 9. 状态管理设计

### 9.1 状态分型

#### S1 持久业务状态

如:

- 当前品牌
- 当前机型
- 当前版本
- 当前已应用预设
- 更新模式
- 是否显示引导

#### S2 瞬时页面状态

如:

- 当前页签
- 滚动位置
- 浮层显示状态

#### S3 编辑器状态

如:

- Dirty
- Undo 栈
- 当前焦点
- 当前编辑模式

#### S4 同步状态

如:

- 最近一次外部修改信号
- 最近一次同步时间
- 是否正在处理外部事件

### 9.2 状态写入约束

禁止:

- 页面直接 `localStorage.setItem`
- 页面直接决定持久配置结构

要求:

- 所有持久状态由 Store 统一写入
- Store 统一触发同步事件
- 渲染层 UI 只能订阅，不直接拼业务状态

---

## 10. 多窗口同步设计

### 10.1 同步目标

必须同步:

- 用户配置
- 当前机型 / 版本
- 当前已应用预设
- 更新模式
- 主题
- 引导开关

只提示不直接覆盖:

- 当前预设内容被其他窗口修改
- 参数页有未保存内容

不自动同步:

- 当前窗口未保存的编辑草稿
- 正在进行中的复杂编辑输入态

### 10.2 同步机制

建议采用统一同步层:

- `CrossWindowSyncService`

同步来源:

- `storage` 事件
- 必要时可升级为 `BroadcastChannel`

### 10.3 冲突策略

- 普通配置: 最后写入者生效
- 当前预设内容: 弹窗提示是否刷新
- 参数编辑草稿: 保留当前窗口草稿，不直接覆盖

---

## 11. 预设系统架构

### 11.1 PresetService 职责

- 解析当前机型和版本对应的预设上下文
- 读取本地预设
- 读取在线清单
- 下载预设
- 应用预设
- 识别当前使用状态
- 识别真实版本
- 统一文件操作

### 11.2 关键原则

- 文件名不作为唯一版本来源
- 真实版本优先读文件内容和 manifest
- 当前应用状态由 “当前上下文键 + 文件路径/文件名” 决定
- 默认资源与用户本地文件必须分层

### 11.3 推荐上下文键

统一上下文键:

- `printerId + versionType`

例如:

- `a1_standard`
- `a1_quick`

禁止出现:

- `a1_null`

这类无版本上下文必须在服务层被拦截。

---

## 12. 参数编辑器架构

### 12.1 ParamEditorService

负责:

- 加载当前预设
- 参数 schema 映射
- 参数视图模型生成
- 保存
- 恢复默认
- 外部变更检测

### 12.2 ParamEditorStore

负责:

- 当前快照
- 已保存快照
- Undo 栈
- Redo 栈
- Dirty 状态
- 当前编辑模式
- 焦点信息

### 12.3 编辑模式

支持双模式:

- 原始文本模式
- 结构化分行模式

两种模式共用同一份底层状态，不允许各自维护独立真相源。

### 12.4 冲突处理

当外部窗口修改当前预设:

- 如果当前窗口无未保存修改，可直接提示刷新
- 如果当前窗口有未保存修改，必须弹窗明确说明风险

---

## 13. 校准系统架构

### 13.1 CalibrationService

负责:

- 读取当前预设中的 x/y/z 偏移
- 写回偏移
- 提供当前偏移摘要
- 判断校准是否可用

### 13.2 关键规则

- 无机型、无版本、无当前预设时，校准必须禁用
- 校准写回后必须广播当前预设变更
- 校准页、参数页、下载页必须围绕同一当前预设上下文刷新

---

## 14. 更新系统架构

### 14.1 UpdateService

负责:

- 读取当前安装版本
- 获取远端 app manifest
- 比较版本
- 触发红点提醒
- 下载补丁
- 应用补丁
- 触发重启

### 14.2 核心原则

- 当前安装版本与远端 manifest 缓存必须隔离
- 远端 manifest 只能作为“远端状态”
- 本地版本判断必须以安装内容为准
- 更新失败必须保留旧版本运行能力

### 14.3 更新模式

- `manual`
- `auto`

默认建议:

- `manual`，即有新版本时红点提醒

---

## 15. 发布系统架构

### 15.1 ReleaseService

负责:

- 版本信息读写
- app manifest 生成
- release notes 生成
- patch 构建
- 安装包构建
- 输出结构校验

### 15.2 DefaultResourceService

负责:

- 默认品牌管理
- 默认机型管理
- 默认图片导入
- 默认预设读写
- presets manifest 管理

### 15.3 发布原则

- 运行时项目与发布系统共享服务层，不共享页面层
- 发布中心优先作为独立开发者工具存在
- 用户端不暴露维护入口

---

## 16. IPC 设计原则

### 16.1 IPC 分类

建议按域划分:

- `catalog:*`
- `preset:*`
- `calibration:*`
- `update:*`
- `release:*`
- `system:*`

### 16.2 IPC 规则

- 一个 handler 只做一个领域动作
- 所有 IPC 输入输出必须有明确结构
- 失败必须返回标准错误对象
- 不允许页面拼接文件路径后随意调用主进程写文件

---

## 17. 日志与错误处理

### 17.1 日志层级

- `INFO`
- `WARN`
- `ERROR`
- `DEBUG`

### 17.2 日志要求

- 每条关键业务操作必须有开始和结果
- 日志要带上下文键
- 更新、预设、保存、回退必须可追踪

### 17.3 错误处理原则

- 用户错误要可理解
- 系统错误要可定位
- 关键错误要有降级路径

---

## 18. 测试架构

### 18.1 建议测试层次

- 单元测试
- 集成测试
- 端到端冒烟测试
- 构建验证测试

### 18.2 高优先级测试对象

- 版本比较逻辑
- manifest 解析
- 预设应用
- 参数保存
- 恢复默认
- Z/XY 写回
- 多窗口同步
- 热更新链路
- 发布中心输出

---

## 19. 发布与回退架构

### 19.1 发布物

- 安装包
- 补丁包
- app manifest
- presets manifest
- 发布说明

### 19.2 发布流程

1. 保存版本信息
2. 校验 schema
3. 校验版本一致性
4. 打包 patch
5. 打包 installer
6. 运行冒烟验证
7. 上传资源
8. 验证更新

### 19.3 回退要求

- 回退必须可以恢复到最近一个稳定版本
- 回退链路不能依赖用户本地手动操作
- 回退必须有日志

### 19.4 发布架构约束

- 发布中心默认从用户安装包剥离
- 用户端只保留更新检查、下载、应用和回退能力
- 发布中心相关依赖、编辑界面、构建脚本不应进入用户运行时包体
- 默认资源维护工具应与最终用户使用路径隔离

---

## 20. 编码与模块治理规范

### 20.1 模块规范

- 单文件超过 800 行必须拆分评审
- 页面层禁止直接写业务文件
- 所有 schema 变更必须带迁移说明
- 所有跨窗口同步键必须统一登记

### 20.2 命名规范

- 文件名使用英文
- 对外显示名单独存储
- 文件路径不依赖显示名称

### 20.3 技术债治理规则

- 不允许重复定义覆盖同名函数作为长期方案
- 不允许把“临时补丁”直接变成正式实现
- 所有热修复都要在后续进入正式重构清单

### 20.4 包体治理规则

- 任何新依赖引入前都要评估对安装包体积的影响
- 非运行时工具脚本不得进入用户端构建白名单
- 调试资源、参考资料、发布脚本、后台管理页面不得打进用户安装包
- 多语言资源默认只保留中文和英文

---

## 21. 从 0 开始的实施顺序

建议严格按以下顺序:

1. 定义 schema 和领域模型
2. 定义 Store 和 Service 边界
3. 定义 IPC 契约
4. 实现文件和远端适配层
5. 实现预设服务
6. 实现当前上下文状态系统
7. 实现校准和参数编辑器
8. 实现更新系统
9. 实现独立发布工具
10. 补齐自动化测试

禁止:

- 先做一堆页面，再回头补状态层
- 先做功能，再回头补 schema
- 先做更新，再回头理清版本源

---

## 22. 架构结论

如果从 0 开始做，MKP SupportE 的正确架构方向应该是:

- 用户端、资源后台、发布系统三域分离
- 服务层统一承载业务
- Store 统一承载状态
- 文件、配置、云端、更新、发布全部显式建模
- 多窗口同步和冲突处理从第一版就纳入设计

这套架构的目标不是让项目“看起来高级”，而是确保后续继续增加机型、预设、参数、更新和发布能力时，系统不会再次退化成难以维护的大脚本工程。
