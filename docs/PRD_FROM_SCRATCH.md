## 2. 产品概述

### 2.1 产品名称

MKP SupportE

### 2.2 产品一句话定义

一款面向 3D 打印用户的专业桌面软件，用于管理支撑面改善预设、进行偏移校准、编辑参数与 G-code、并通过可靠的版本与发布系统持续交付默认资源与软件更新。

### 2.3 产品本质

从产品本质上看，MKP SupportE 不是单一工具，而是由三套系统组成:

1. 用户端桌面应用
2. 默认资源管理后台
3. 版本发布与热更新系统

补充约束:

- 三套系统不等于必须打进同一个用户安装包
- 面向用户交付时，用户安装包只包含用户端桌面应用和运行必需资源
- 默认资源后台与版本发布系统可以是独立开发者工具

### 2.4 产品边界

本产品负责:

- 机型与版本选择
- 在线和本地预设管理
- 偏移校准
- 预设参数编辑
- 默认资源管理
- 软件与预设更新
- 发布打包与版本控制

本产品不负责:

- 切片功能本身
- 3D 模型设计
- 云端账号系统
- 多人协作权限系统
- 完整云端数据库后台
- 用户画像和行为分析平台
- 多语言平台化扩展

### 2.5 产品硬约束

- Windows 安装包体积必须控制在 `99MB` 以内
- 只支持中文和英文界面
- 仅保留基础错误日志，不采集用户画像类数据
- 发布中心不进入普通用户默认安装包

## 3. 用户与场景

### 3.1 核心用户类型

#### U1 普通 3D 打印用户

特征:

- 希望通过图形界面完成设置
- 不愿直接编辑 JSON 或 G-code
- 关注打印稳定性和效果

核心诉求:

- 快速找到自己的机型和版本
- 能下载和应用正确预设
- 能完成校准
- 出错时有明确提示

#### U2 进阶调参用户

特征:

- 会反复调整预设参数
- 会查看和修改 G-code
- 对撤销、恢复默认、版本一致性要求高

核心诉求:

- 参数结构清晰
- 改动可撤销
- 修改安全
- 不会因为切页面或多窗口而丢失上下文

#### U3 维护者 / 发布者

特征:

- 维护默认机型目录
- 维护默认预设
- 负责版本更新和补丁打包

核心诉求:

- 发布流程简单稳定
- 默认资源可直接维护
- 打包结果可验证
- 更新链路可回退

## 4. 用户核心问题

### 4.1 普通用户问题

- 不知道该选哪个机型和版本
- 不知道自己当前是否已经应用了正确预设
- 不知道 Z/XY 偏移怎么调
- 不会手动修改 JSON 文件
- 更新后不确定是否成功

### 4.2 进阶用户问题

- 参数很多，不知道哪些关键
- 一次修改多个参数后，怕保存错
- 撤销和恢复默认不够可靠
- 多窗口或多页面切换时状态容易乱

### 4.3 维护者问题

- 默认机型和图片维护分散
- 默认预设修改成本高
- app manifest、patch、安装包流程容易混
- 版本说明、补丁、完整包没有统一约束

## 5. 产品目标

### 5.1 总目标

建立一套专业、稳定、可持续维护的桌面软件体系，使用户能够在不直接接触底层 JSON 和 G-code 的情况下，完成机型选择、预设管理、校准和参数调整，同时让维护者能够稳定管理默认资源并完成发布。

### 5.2 分目标

#### G1 新用户可快速完成第一次使用

衡量标准:

- 10 分钟内完成机型选择、预设应用、一次基础校准

#### G2 高级用户能安全地修改参数

衡量标准:

- 参数修改支持保存、撤销、恢复默认、未保存拦截、外部修改提示

#### G3 维护者能稳定发版

衡量标准:

- manifest、补丁、安装包、发布说明统一生成并通过校验

#### G4 软件具备长期演进能力

衡量标准:

- 模块边界清晰
- 数据模型稳定
- 发布与回退链路可验证

## 7. 核心用户流程

### 7.1 普通用户首次使用流程

1. 打开软件
2. 新手引导帮助理解流程
3. 选择品牌与机型
4. 选择版本类型
5. 下载或应用本地预设
6. 进入校准页面完成 Z/XY 调整
7. 进入参数页做进一步微调
8. 保存
9. 配置切片后处理脚本

### 7.2 进阶用户调参流程

1. 打开已使用的机型与预设
2. 查看当前参数
3. 修改普通参数或 G-code 行
4. 撤销 / 重做多次对比
5. 保存所有修改
6. 如有问题，恢复默认

### 7.3 维护者发版流程

1. 打开发布中心
2. 编辑版本号、日期、更新内容
3. 选择热更新或完整构建模式
4. 执行打包
5. 校验输出
6. 上传补丁 / 安装包
7. 验证更新链路

## 9. 非功能需求

### 9.1 稳定性

- NFR-1 启动不崩溃
- NFR-2 主流程不因资源缺失直接空白
- NFR-3 配置写入具备基本容错

### 9.2 可维护性

- NFR-4 页面逻辑、业务逻辑、文件逻辑分层
- NFR-5 不允许超大职责文件长期增长
- NFR-6 数据结构必须有 schema

### 9.3 一致性

- NFR-7 多页面状态一致
- NFR-8 多窗口状态一致
- NFR-9 当前预设状态全局一致

### 9.4 可恢复性

- NFR-10 参数支持撤销
- NFR-11 支持恢复默认
- NFR-12 更新失败支持回退

### 9.5 发布可控性

- NFR-13 manifest、patch、安装包有统一口径
- NFR-14 打包结果可校验
- NFR-15 版本号与资源一致

### 9.6 性能与体验

- NFR-16 页面切换流畅
- NFR-17 侧边栏收起展开不卡顿
- NFR-18 长列表可接受
- NFR-19 编辑器操作反馈明确

### 9.7 平台与兼容

- NFR-20 Windows 路径兼容
- NFR-21 中文路径兼容
- NFR-22 打包后运行依赖完整

## 11. 验收标准

### 11.1 用户端验收标准

- 用户能够从空状态完成一次完整预设应用和校准
- 当前机型、版本、预设在页面切换和多窗口中保持一致
- 参数保存后校准页数值同步
- 外部窗口修改当前预设时，当前窗口会提示是否刷新
- 恢复默认后可以回到默认预设内容

### 11.2 发布端验收标准

- 能编辑并保存版本信息
- 能生成热更新补丁
- 能生成完整安装包
- 构建输出可检查
- app manifest 与 package version 一致

## 13. 风险与约束

### 13.1 主要风险

- 状态同步复杂
- 预设编辑器复杂度高
- 默认资源与用户覆盖层边界容易混乱
- 更新系统出错影响信任
- 发布中心与用户端逻辑耦合

### 13.2 产品约束

- 必须保留桌面软件形态
- 不依赖 WebView2 路线重构
- 必须支持离线基本可用
- 必须支持热更新与完整安装包双链路
- 默认仅提供中英文
- 不建设云端用户分析系统
- 仅保留基础日志诊断系统
- 发布后台必须可独立部署，不进入用户默认安装包

---

## 14. 从 0 开始的产品策略结论

如果从 0 开始做，正确策略不是先把界面铺满，而是:

1. 先冻结边界
2. 先定义核心闭环
3. 先建立数据模型
4. 再搭架构与状态层
5. 最后再做页面与体验

只有这样，MKP SupportE 才能从“功能很多的工具”真正成长为“可持续维护的专业桌面软件”。
